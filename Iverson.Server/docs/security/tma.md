## 2. System overview

Iverson is a generic, schema-driven, multi-store entity platform. Clients register a typed schema over gRPC (`RegisterSchema`), then read/write/search entity data through four gRPC services; the server fans each write out from Postgres (system of record) through Kafka to StarRocks (analytics/search projection) and Qdrant (vector/chunk projection). It matters from a security standpoint because it is a *shared, multi-tenant-shaped* data platform ŌĆö the schema a client registers determines table/column/collection structure across three different datastores, and a custom row/field-level authorization layer (on top of standard JWT authentication) is the only thing standing between one caller's data and another's.

## 3. Architecture & data flows

**Runtime**: a single ASP.NET Core process (`Iverson.Api`) runs as either `api` (gRPC services + HTTP health/admin) or `worker` (Kafka consumers + background reconciliation), selected by `WORKLOAD_ROLE`.

**Write path**: client ŌåÆ gRPC `Post`/`Update` (protobuf `Struct`) ŌåÆ row/field authorization check ŌåÆ Postgres write (system of record, in the same transaction as an outbox row) ŌåÆ opportunistic Kafka publish of an `EntityEvent` (JSON, no schema registry) ŌåÆ `worker`-role consumers (`EngagementStoreConsumer` ŌåÆ StarRocks, `IntelligenceStoreConsumer` ŌåÆ Qdrant) project the same event into the other two stores, independently of the original write-time authorization decision.

**Read path**: client ŌåÆ gRPC `Get`/`GetMany`/`Search`/`Aggregate`/`GroupBy`/`Pipeline`/`SearchSimilar`/`SearchChunks` ŌåÆ row/field authorization check (ownership clause + field masking) ŌåÆ routed to Postgres (key-based), StarRocks (search/aggregate SQL), or Qdrant (vector/chunk search), depending on RPC and schema shape.

## 4. Threat actors

| Actor | Capabilities | Motivation | In scope? |
|---|---|---|---|
| **External opportunistic** | Internet-reachable scanning/probing, no valid credentials | Reconnaissance, indiscriminate exploitation | Yes, but narrow ŌĆö only the anonymous surfaces (H, and Authentik's own login page) are reachable without a credential; everything else requires a valid JWT. |
| **External targeted / holder of a low-privilege credential** | A legitimate JWT for **any** of the 4 configured audiences (e.g. a signed-up user of the human-facing web app, or a leaked/compromised `webtest`/`loadtest` client secret) | Data theft, privilege escalation, disruption | **Yes ŌĆö primary actor for this model.** Because RPC-level authentication does not differentiate audiences, this actor's reach is much broader than "low-privilege" would suggest (see ┬¦6 F1/F2). |
| **Insider ŌĆö privileged operator** | Cluster/kubeconfig access, DB credentials, Helm access | Accidental misconfiguration, occasionally malicious | Yes, but different blast radius ŌĆö this actor is already trusted with direct store access; relevant mainly for defense-in-depth (┬¦6 F4) rather than a novel path. |
| **Insider ŌĆö regular authenticated app user** | Same as "external targeted" above once logged in via the human-oidc-client audience | Curiosity, unauthorized access to other tenants'/users' data | Yes ŌĆö effectively the same actor as "external targeted" in this system, since there is no separate privilege tier between "logged-in user" and "service client." |
| **Compromised supply chain** | Malicious/compromised NuGet dependency | Backdoor, data exfiltration | Out of scope for TMA ŌĆö dependency CVE/integrity scanning is `critical-security-review`'s job. Dependencies observed are mainstream (Npgsql, Dapper, Confluent.Kafka, Qdrant.Client, MySqlConnector) with no anomalies noted during recon. |
| **Automated bots (credential stuffing, etc.)** | Scripted login attempts against Authentik | Account takeover | Largely an Authentik product concern, not this codebase; noted but not scored here. |

## 5. Threat analysis (STRIDE-per-element)

### A. Schema management (`RegisterSchema`)
- **Spoofing**: mitigated ŌĆö requires a valid JWT for one of the 4 audiences (global `RequireAuthenticatedUser()` fallback policy).
- **Tampering**: **not mitigated.** Client-controlled `TypeName`/property names flow through `ToSnakeCase()` (case transformation only, no character-class filtering) into DDL identifier positions, string-interpolated and double-quote/backtick-delimited, in both Postgres (`PostgresSchemaManager.cs:39-47`) and StarRocks (`StarRocksSchemaManager.cs:56`). See F2.
- **Repudiation**: schema changes are logged (`_logger.LogInformation("[RegisterSchema] root=...")`) but with no actor identity attached beyond what's in the ambient trace context ŌĆö insufficient for incident forensics on its own.
- **Information disclosure**: none direct; secondary via authorization-rule tampering below.
- **Denial of service**: a caller can register unboundedly many schemas/tables; no rate limit or quota observed.
- **Elevation of privilege**: **not mitigated ŌĆö the core finding.** `RegisterSchema` never calls `IRowFieldAuthorizationEvaluator` and has no `Operator`-policy gate (unlike the HTTP admin endpoints, which do). `SchemaRegistry.RegisterAsync` performs an unconditional blind dictionary replace (`_schemas[descriptor.TypeName] = descriptor`) with no merge against a prior descriptor's `Authorization` rules. See F1.
- **Existing mitigations**: authentication (any valid JWT), and narrow semantic validation (owner-field must be a declared string-typed scalar column, chunk-field key-collision check) ŌĆö these harden the *shape* of a schema but not *who* may define it.
- **Residual risk**: Critical, and it is the single highest-leverage element in this model ŌĆö everything in element B is only as trustworthy as the `Authorization` rules element A allows any of the 4 audiences to set.

### B. CRUD / retrieval / search RPCs
- **Spoofing**: mitigated (JWT required); acting-user identity is a second, independently-validated JWT (`ActingUser` scheme, own `ValidAudiences`), not merely a claim inside the first token.
- **Tampering**: field-level write rejection (`AuthorizationFieldMasking.RejectDisallowedFields`) throws `InvalidArgument` for any field the caller's role isn't permitted to write; owner field is force-set (not client-settable) for ownership-required callers, and is immutable after creation for existing rows. Well-mitigated *given* correct `Authorization` rules ŌĆö but those rules are only as trustworthy as element A.
- **Repudiation**: writes carry a `TraceId`; no explicit actor-identity audit log observed beyond ambient logging/OTel spans.
- **Information disclosure**: field masking on read (`AuthorizationFieldMasking.MaskDisallowedFields`) removes disallowed fields from responses; row-level ownership filtering is applied as a `WHERE`/`Filter.Must` clause at the store layer, not just post-filtered in memory. Fails closed: `RowFieldAuthorizationEvaluator.Evaluate` returns `Denied=true` both when a schema has no `Authorization` rules configured *and* when the acting-user JWT is absent/unparseable (verified directly in `RowFieldAuthorizationEvaluator.cs:11-15`) ŌĆö omitting the acting-user header is not a bypass, it is a hard deny for any schema that requires it.
- **Denial of service**: `Search`/`Aggregate`/`GroupBy`/`Pipeline` accept a caller-driven filter/aggregation DSL translated into StarRocks SQL ŌĆö no query-cost/complexity limit observed (e.g. unbounded clause count, unbounded page size) beyond the language-level validation described below.
- **Elevation of privilege**: `Search`/`Aggregate`'s `AggregationSpec.Expression`/`MetricSpec.Expression` are validated with a **word-token allow-list only** (no punctuation/comment denylist), while the structurally similar `Pipeline`'s `DeriveColumn.Expr` additionally blocks `; ' `` -- /* */` (`StarRocksPipelineBuilder.cs:320-334`). This inconsistency between two validation paths for conceptually the same kind of caller-supplied expression is worth closing, though full exploitability requires code-level verification beyond TMA's scope. See F7.
- **Existing mitigations**: comprehensive row/field evaluator coverage across all four gRPC services' data-bearing RPCs; `SearchSimilar`/`SearchChunks`'s Qdrant filters are validated against a known-column allow-list before being translated by a typed filter builder (no raw string injection surface there).
- **Residual risk**: Low-to-Medium *assuming* element A is fixed; the expression-validation inconsistency (F7) is a smaller, independent residual.

### C. Acting-user identity propagation
- **Spoofing**: the acting-user JWT is validated independently (own signing-key/audience checks), not merely trusted from a claim in the primary token ŌĆö a caller cannot forge an acting-user identity by manipulating the primary JWT alone.
- **Tampering**: N/A (identity data, not being written).
- **Repudiation**: acting-user `sub` becomes `OwnerValue` for row comparisons ŌĆö this is the actual audit-relevant identity for authorization decisions, and it is only present when the header is supplied.
- **Information disclosure**: `MapInboundClaims=false` is set to preserve the raw `sub` claim (needed for ownership comparison) rather than ASP.NET's default claim-type remapping ŌĆö a deliberate, narrow choice, not a leak.
- **Denial of service**: N/A.
- **Elevation of privilege**: none found ŌĆö see B above; absence of the header fails closed, not open.
- **Existing mitigations**: dual independent JWT validation; fail-closed evaluator behavior.
- **Residual risk**: Low.

### D. Kafka event bus + projection consumers
- **Spoofing**: N/A ŌĆö internal topic, not externally reachable (no ingress exposes Kafka).
- **Tampering**: `EntityEvent`s are plain JSON with no signing/HMAC; any producer with network access to the internal Kafka listener could inject a fabricated event that `IntelligenceStoreConsumer`/`EngagementStoreConsumer` would project verbatim, including an arbitrary `ownerValue` (`IntelligenceStoreConsumer.cs:138-140` reads `ownerValue` straight from the untrusted-by-the-consumer JSON payload). This is not independently reachable by an external caller (Kafka has no external ingress) ŌĆö its real relevance is as a downstream amplifier of A/B: an attacker who achieves an elevated write identity via F1 can stamp arbitrary ownership data that then propagates unchecked into two of the three stores' read-time filtering. See F3.
- **Repudiation**: no re-attribution at consume time; the original write-time actor identity is not part of the event schema.
- **Information disclosure**: N/A beyond what already crossed the write-time boundary.
- **Denial of service**: reconciliation/DLQ machinery (`ReconciliationQueueWorker`, `DlqMonitorConsumer`) exists specifically to handle backlog/failure scenarios ŌĆö reasonable resilience posture.
- **Elevation of privilege**: none independently; amplifies F1 as described.
- **Existing mitigations**: Kafka itself is not externally reachable in either deployment target; TLS+SCRAM in kind/cloud restricts who can produce/consume even on the internal network.
- **Residual risk**: Medium, chained to F1/F2 ŌĆö see F3.

### E. Data stores at rest
- **Spoofing / Tampering / Info disclosure**: **Qdrant has no authentication in either deployment target** (compose or kind/cloud) ŌĆö reachability is gated solely by network topology (Docker's flat network in compose; Kubernetes `NetworkPolicy` default-deny + per-service allow rules in kind/cloud). Any workload that can reach its ClusterIP ŌĆö a compromised sidecar, a misconfigured NetworkPolicy, or literally any container in the compose topology ŌĆö has full unauthenticated read/write. StarRocks, by contrast, **does have real authentication in kind/cloud**: `deploy/helm/iverson/charts/starrocks/templates/job-create-user.yaml` provisions a dedicated `iverson_app` user with a Helm-generated password and scoped `GRANT`s, and `Iverson.Api`'s deployment wires that password into `ConnectionStrings__StarRocks` (`charts/api/templates/deployment.yaml:85-91`) ŌĆö confirmed directly in both files, not assumed. StarRocks in **compose** still has no auth (root, empty password) ŌĆö that target's posture is unchanged. Postgres has authentication (SCRAM) and TLS (kind/cloud only, via CNPG). See F4 (rescoped to Qdrant-only, and to StarRocks-in-compose-only).
- **Repudiation**: no store-level audit logging observed for direct access (expected, given no auth exists to attribute to).
- **Denial of service**: not specifically assessed (resource quotas are an operational/arch-review concern, not scored here).
- **Elevation of privilege**: bypassing the gRPC layer entirely and connecting directly to Qdrant (either target) or StarRocks (compose only ŌĆö kind/cloud requires the `iverson_app` credential) is a full authorization bypass by construction, since all authorization enforcement lives in the API layer, not the stores.
- **Existing mitigations**: kind/cloud's default-deny `NetworkPolicy` is the sole control; Postgres has its own auth+TLS as a second, independent layer.
- **Residual risk**: High in kind/cloud for Qdrant specifically (single control point, no auth backstop); StarRocks in kind/cloud is no longer a gap (real auth confirmed in place). The compose posture (both stores unauthenticated) is explicitly dev-only and not scored at production severity.

### F. Authentik IdP & OAuth2 client credentials
- **Spoofing**: Authentik itself is the trust anchor for every JWT in this system; compromising it is a full-system compromise. Not independently re-assessed here (product-level Authentik security is out of scope; the API's *use* of Authentik's tokens is in scope and covered above).
- **Tampering / Info disclosure**: compose-target OAuth2 client secrets and default user passwords are static, human-readable, and committed to source control (`deploy/helm/iverson/charts/authentik/blueprints/compose-only/service-clients.yaml`), clearly labeled `dev-only-not-for-production`. Kind/cloud generates real per-release secrets via Helm `randAlphaNum`, persisted with `helm.sh/resource-policy: keep`. See F5.
- **Denial of service**: N/A.
- **Elevation of privilege**: none beyond what's already covered by the audience/authorization discussion above.
- **Existing mitigations**: kind/cloud's generated-secret path is sound; the compose path is explicitly a local-dev convenience, not a production posture.
- **Residual risk**: Medium, bounded to the compose/local-dev target.

### G. HTTP admin/operator endpoints
- **Spoofing**: mitigated (JWT required).
- **Elevation of privilege**: mitigated ŌĆö `RequireAuthorization("Operator")` gates `/admin/reconcile/{typeName}`, `/admin/dlq`, `/admin/dlq/{id}/replay`, checked against a `groups`/`scope` claim (`OperatorAuthorizationPolicy.cs:9-15`). This is the pattern element A (`RegisterSchema`) conspicuously lacks.
- **Residual risk**: Low ŌĆö this element is a model of what A should look like, not a gap itself.

### H. HTTP health/metrics endpoints
- **Information disclosure**: Prometheus scrape and `/health`/`/probe/*` are `AllowAnonymous` by design; metrics could reveal operational details (queue depths, error rates) to an unauthenticated observer, but this is a conventional, low-sensitivity tradeoff for operability, not a distinct finding.
- **Denial of service**: unauthenticated endpoints are a nonzero DoS surface but a standard one (rate-limiting/ingress-level concern, not application-specific).
- **Residual risk**: Low, accepted by design.

### I. Ollama embedding integration
- **Spoofing / Tampering**: no authentication on the Ollama HTTP call; relies entirely on network-level trust (ClusterIP-only, not internet-reachable). Consistent with Ollama's own lack of a first-class auth story.
- **Residual risk**: Low, given no external reachability and no sensitive-data-uniquely-at-risk here beyond what already crossed the write-time boundary (the text being embedded is the same entity data already authorized at write time).

### J. Deployment/network posture
- **Tampering / Info disclosure / Elevation**: compose is explicitly a flat, plaintext, all-ports-published dev topology ŌĆö not scored at production severity. Kind/cloud has default-deny NetworkPolicy, TLS on ingress/Kafka/Postgres, and Helm-generated secrets ŌĆö a meaningfully different (better) posture, with the residual gaps already called out under E (Qdrant/StarRocks no-auth) and F (AuthentikŌåöAPI plaintext HTTP even in kind/cloud, noted for completeness but bounded by NetworkPolicy since it's internal-only traffic).
- **Residual risk**: reflects the union of E/F above; no independent finding here beyond what's already captured.

## 6. Findings ŌĆö prioritized roadmap

| # | Severity | Tag | Finding |
|---|---|---|---|
| **F1** | **Critical** | Implemented | `RegisterSchema` (`Grpc/ObjectMappingGrpcService.cs:45-129`) never invokes `IRowFieldAuthorizationEvaluator` and has no `Operator`-style policy gate ŌĆö unlike the HTTP admin endpoints, which do. Any caller holding a valid JWT for **any** of the 4 configured audiences (including the human-facing web app's own OIDC client) can register a schema for an existing `TypeName`, and `SchemaRegistry.RegisterAsync` (`Schema/SchemaRegistry.cs:47-54`) performs an unconditional blind replace with no merge against the prior descriptor's `Authorization` rules. **Concrete attack**: attacker submits a `SchemaRequest` for a sensitive existing entity type whose `Authorization.RowPermissions` grants their own JWT `groups` claim `CanReadAll`/`CanWriteAll`/`CanDeleteAll = true`; the new rules take effect immediately for every subsequent call through elements B (all four gRPC services share the same evaluator). This is a direct, low-complexity, valid-but-unprivileged-credential elevation of privilege ŌĆö no chaining required beyond authenticating once. **Fix direction**: gate `RegisterSchema` behind an `Operator`-equivalent policy (mirroring `OperatorAuthorizationPolicy`), and/or require re-registration of an existing `TypeName` to preserve or explicitly re-affirm its prior `Authorization` rules rather than silently discarding them. |
| **F2** | **Critical** | Implemented | Client-supplied `TypeName`/property names are used as SQL/DDL identifiers with no character-class validation ŌĆö `NamingExtensions.ToSnakeCase()` (`Api/NamingExtensions.cs:5-14`) performs only case transformation (insert `_` before uppercase, lowercase everything), passing through any other character unchanged. Confirmed unescaped, string-interpolated use in `CREATE TABLE IF NOT EXISTS "{schema.TableName}"` / `ALTER TABLE "{schema.TableName}"` (Postgres, `Iverson.Sql/PostgresSchemaManager.cs:39-47,53-72`) and the equivalent backtick-delimited pattern in StarRocks (`Iverson.StarRocks/StarRocksSchemaManager.cs:56,99,112`), both executed via a raw `ExecuteAsync` with no parameterization possible for DDL identifiers. **Concrete attack**: same reachability as F1 (any valid JWT, since `RegisterSchema` has no authorization gate) ŌĆö a `TypeName` containing a literal `"` (Postgres) or `` ` `` (StarRocks) plus additional SQL breaks out of the identifier-quoting context, executing arbitrary DDL/DML with whatever privileges the application's database role holds. Verified this does **not** extend to Qdrant: `QdrantCollectionManager.cs:30` passes the collection name as a plain string argument to the Qdrant client's typed gRPC API, not a textual query language ŌĆö not susceptible to the same injection class. **Fix direction**: allow-list identifier characters (e.g. `^[A-Za-z0-9_]+$`) on `TypeName` and all property names at `RegisterSchema` time, rejecting anything else with `InvalidArgument`, before any descriptor reaches a schema manager. |
| **F3** | **High** | Implemented | `IntelligenceStoreConsumer` (`Consumers/IntelligenceStoreConsumer.cs:138-140`) and the equivalent StarRocks consumer trust the Kafka-carried `EntityEvent`'s embedded field values ŌĆö including the value later used for ownership filtering ŌĆö verbatim, with no re-authorization at consume time. Not independently reachable (Kafka has no external ingress in either deployment target), but it means the two projected stores' read-time ownership filtering is only as trustworthy as whatever produced the original event ŌĆö directly amplifying F1/F2 (an attacker who achieves an elevated write identity can stamp arbitrary ownership data that propagates unchecked). **Fix direction**: either re-derive/re-validate the owner value at projection time from the authoritative Postgres row rather than trusting the event payload, or accept this as a documented single-point-of-enforcement design decision ŌĆö but document it as a decision, not an oversight. |
| **F4** | **High** (Qdrant, both targets) / **Medium** (StarRocks, compose only) | Planned | **Qdrant has no authentication in either deployment target**; reachability is gated solely by network topology (`deploy/helm/iverson/templates/networkpolicies.yaml`'s default-deny + allow-rules in kind/cloud, or Docker's flat network in compose) ŌĆö a single point of failure with no auth backstop. **StarRocks in kind/cloud already has real authentication** (`iverson_app` user + Helm-generated password + scoped `GRANT`s, wired into `ConnectionStrings__StarRocks` ŌĆö `charts/starrocks/templates/job-create-user.yaml`, `charts/api/templates/deployment.yaml:85-91`), confirmed directly in code ŌĆö this was incorrectly scoped as a gap in the first draft of this document. StarRocks in **compose** still has no auth (root, empty password), consistent with that target's explicitly dev-only posture. Postgres has both TLS and SCRAM auth in kind/cloud as a second independent layer. **Fix direction**: give Qdrant its own authentication layer (API key) as defense-in-depth behind the NetworkPolicy boundary in kind/cloud; StarRocks needs no further work there ŌĆö only compose's StarRocks posture remains an accepted dev-only gap. |
| **F5** | **Medium** | Implemented | Compose-target OAuth2 client secrets and default user passwords are static, human-readable, and committed to source control (`deploy/helm/iverson/charts/authentik/blueprints/compose-only/service-clients.yaml`), clearly labeled `dev-only-not-for-production...`. Blast radius is bounded to the local/compose target ŌĆö kind/cloud generates real per-release secrets via Helm. Flagged because "credentials in git," even clearly-labeled dev ones, is worth a deliberate decision rather than an implicit one (e.g. confirming no compose deployment is ever exposed beyond localhost). **Fix direction**: no change needed if compose is confirmed local-only in every real deployment; if compose is ever used in a shared/CI environment, these should move to generated-at-deploy-time secrets like the kind/cloud path already does. |
| **F6** | **Medium** | Planned | There is no method- or audience-level authorization tier above row/field authorization ŌĆö all 4 JWT audiences (including the human-facing web app's own users) get identical RPC-level access to schema management, CRUD, search, and aggregation. Row/field authorization (element B) is well-built, but it was evidently never extended to cover schema-administration operations (F1 is the concrete symptom). **Fix direction**: introduce a distinct authorization tier for schema-administration RPCs (e.g. a dedicated scope/audience/role, checked independently of row/field rules), so that adding new elevated-privilege operations in the future doesn't silently inherit "any authenticated caller" by default the way `RegisterSchema` did. |
| **F7** | **Medium** | Planned | `Search`/`Aggregate`'s `AggregationSpec.Expression`/`MetricSpec.Expression` fields are validated with a word-token allow-list only, while the structurally similar `Pipeline.DeriveColumn.Expr` additionally blocks `; ' `` -- /* */` (`StarRocksPipelineBuilder.cs:320-334`). Two validation paths for the same conceptual input (caller-supplied expression flowing into StarRocks SQL construction) shouldn't diverge in strictness. Full exploitability of the weaker path was not independently verified here ŌĆö that level of confirmation belongs to `critical-security-review`. **Fix direction**: align `Expression` validation with `DeriveColumn.Expr`'s stricter blacklist (or better, replace both with a shared allow-list-based expression parser), and have CSR verify exploitability as a follow-up. |

## 7. When to refresh

Refresh this document when any of the triggers in this skill's Appendix C fire ŌĆö most relevant for this system: a change to the authorization model (e.g. if F1/F6 are fixed by introducing a new schema-admin authorization tier ŌĆö re-verify the fix closes the gap rather than just re-describing it), a new external integration, a new trust boundary (e.g. multi-tenancy beyond the current owner-field model), or a major architectural change (e.g. Qdrant gaining native auth, or compose's StarRocks posture changing, either of which would materially change F4). Otherwise, treat this as a quarterly check-in artifact, or refresh immediately after any incident touching schema registration, the acting-user auth layer, or direct data-store access.
