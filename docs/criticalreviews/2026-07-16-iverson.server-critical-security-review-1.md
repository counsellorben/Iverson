## Finding #1: `RegisterSchema` has no authorization check — any of 4 low-privilege audiences can redefine any entity's access rules

**Vulnerability:** Broken Access Control — A01 (Missing Function-Level Authorization / Privilege Escalation)
**Severity:** Critical
**Confidence:** Confirmed
**Attack Complexity:** Low
**TMA cross-link:** Confirms TMA #F1 with code-level evidence

**Location:**
- File: `Iverson.Api/Grpc/ObjectMappingGrpcService.cs`, Lines: 45–129 (`RegisterSchema`)
- Related: `Iverson.Api/Schema/SchemaRegistry.cs:47-54` (`RegisterAsync` — blind replace), `Iverson.Api/Program.cs:117-128` (4 configured JWT audiences)

**Risk & Exploit Path:**
Every gRPC method authenticates via the global fallback policy (`RequireAuthenticatedUser()`, `Program.cs:136-138`), which accepts a valid JWT for **any** of 4 configured audiences: `human-oidc-client` (the human-facing web app), `loadtest-client`, `webtest-client`, `admin-automation-client` (`deploy/helm/iverson/charts/api/templates/deployment.yaml:117-128`). Unlike the HTTP admin endpoints (`/admin/*`, gated by `RequireAuthorization("Operator")`), `RegisterSchema` performs zero authorization check beyond that baseline authentication — it never references `IRowFieldAuthorizationEvaluator`, and there is no policy/role check of any kind in the method body.

`SchemaRegistry.RegisterAsync` (`Schema/SchemaRegistry.cs:54`) is an unconditional dictionary assignment — `_schemas[descriptor.TypeName] = descriptor` — with no merge against a previously-registered descriptor's `Authorization` rules.

**Exploit:** An attacker holding a JWT for any of the 4 audiences (including a signed-up user of the human-facing web app — this is not a service-only credential) calls `RegisterSchema` for an existing, sensitive `TypeName`, supplying a `SchemaRequest` whose `Authorization.RowPermissions` grants their own JWT `groups` claim `CanReadAll = CanWriteAll = CanDeleteAll = true`. The new rules take effect immediately and apply identically across all four gRPC services (`ObjectMapping`, `ObjectPersistence`, `ObjectRetrieval`, `ObjectSearch`), since they all resolve authorization through the same `SchemaRegistry` entry. No further chaining, timing, or privileged position is required — one authenticated call is sufficient.

**Evidence / Trace:**
```csharp
// ObjectMappingGrpcService.cs:45-59 — no IRowFieldAuthorizationEvaluator reference anywhere in this method
public override async Task<SchemaResponse> RegisterSchema(
    SchemaRequest request, ServerCallContext context)
{
    ...
    foreach (var typeDesc in new[] { request.RootType }.Concat(request.Dependents))
    {
        var descriptor = SchemaBuilder.BuildDescriptor(typeDesc, _embedding);
        // ← VULNERABLE: descriptor.Authorization comes straight from client-supplied typeDesc,
        //   no check of who is allowed to set/change it
        ...
        await _registry.RegisterAsync(descriptor);   // ← VULNERABLE: blind replace, see below
```
```csharp
// SchemaRegistry.cs:47-54
public async Task RegisterAsync(SchemaDescriptor descriptor)
{
    ...
    _schemas[descriptor.TypeName] = descriptor;   // ← VULNERABLE: unconditional replace, no merge
}
```
Compare to the pattern that correctly exists elsewhere in the same codebase:
```csharp
// Program.cs:139-142 — the pattern RegisterSchema conspicuously lacks
options.AddPolicy("Operator", policy => policy.RequireAssertion(context =>
    OperatorAuthorizationPolicy.IsSatisfiedBy(
        context.User.FindAll("groups").Select(c => c.Value),
        context.User.FindFirst("scope")?.Value)));
```

**Remediation:**
- **Primary fix:** Gate `RegisterSchema` behind an `Operator`-equivalent policy (reuse `OperatorAuthorizationPolicy` or a new dedicated `SchemaAdmin` policy), enforced via `[Authorize(Policy = "...")]` or an explicit in-method check, matching the pattern already used for `/admin/*`.
- **Architectural improvement:** Introduce a distinct authorization tier for schema-administration RPCs, separate from row/field authorization (which is correctly built and well-covered for *data* operations but was evidently never extended to *schema* operations). Additionally, make re-registration of an existing `TypeName` preserve its prior `Authorization` rules unless the caller is specifically privileged to change them, rather than silently discarding them on every call.
- **Defense-in-depth:** Audit-log `RegisterSchema` calls with the calling identity's `sub`/`groups` (currently logs only `TypeName`/dependent count, not the caller), so a future incident can at least reconstruct who registered what.

## Finding #2: Client-controlled `TypeName` flows unescaped into DDL identifiers (Postgres, StarRocks)

**Vulnerability:** SQL/DDL Injection — A03
**Severity:** Critical
**Confidence:** Confirmed
**Attack Complexity:** Low (identifier breakout) / Medium (full multi-statement exploitation, driver-protocol-dependent)
**TMA cross-link:** Confirms TMA #F2 with code-level evidence; narrows scope (Qdrant refuted — see below)

**Location:**
- File: `Iverson.Api/NamingExtensions.cs`, Lines: 5–14 (`ToSnakeCase`)
- File: `Iverson.Sql/PostgresSchemaManager.cs`, Lines: 39–47, 53–72 (`CREATE TABLE`/`ALTER TABLE`)
- File: `Iverson.StarRocks/StarRocksSchemaManager.cs`, Lines: 56, 99, 112 (`CREATE TABLE`/`ALTER TABLE`/`CREATE DATABASE`)

**Risk & Exploit Path:**
Reachable via the same unauthenticated-by-role path as Finding #1 (`RegisterSchema` accepts any of the 4 audiences). `NamingExtensions.ToSnakeCase()` performs only case transformation (insert `_` before an uppercase letter, lowercase everything) — it does not reject, strip, or escape any character. The resulting `TableName` is then string-interpolated directly into DDL, double-quote-delimited in Postgres and backtick-delimited in StarRocks, and executed via Dapper's `ExecuteAsync` with no parameterization possible for identifier positions.

**Exploit path:** a `TypeName` containing a literal `"` (targeting Postgres) or `` ` `` (targeting StarRocks) breaks out of the identifier-quoting context. For Postgres specifically, Npgsql's simple query protocol — used automatically by `NpgsqlCommand`/Dapper when a command has no bound parameters, which is the case for every DDL string built by `PostgresSchemaManager` — permits semicolon-separated multi-statement execution in a single call, meaning a well-formed injected payload can append an entirely separate SQL statement (not just corrupt the current one) with whatever privileges the application's Postgres role holds.

**Evidence / Trace:**
```csharp
// NamingExtensions.cs:5-14 — case transform only, no character-class validation
internal static string ToSnakeCase(this string name)
{
    var sb = new System.Text.StringBuilder();
    for (var i = 0; i < name.Length; i++)
    {
        if (char.IsUpper(name[i]) && i > 0) sb.Append('_');
        sb.Append(char.ToLowerInvariant(name[i]));   // ← every other character passes through unchanged
    }
    return sb.ToString();
}
```
```csharp
// PostgresSchemaManager.cs:39-47
var ddl = $"""
    CREATE TABLE IF NOT EXISTS "{schema.TableName}" (   // ← VULNERABLE: raw interpolation, no escaping of embedded '"'
        {keySql},
        {string.Join(",\n    ", colDefs)}
    )
    """;
...
await conn.ExecuteAsync(ddl);   // ← no parameters bound; Npgsql uses the simple query protocol here
```
```csharp
// StarRocksSchemaManager.cs:56 — identical pattern, backtick-delimited
CREATE TABLE IF NOT EXISTS `{schema.TableName}` (
```

**Scope refinement (Qdrant — refutes the extension TMA hedged on):** `Iverson.Vector/QdrantCollectionManager.cs:30` (`client.CreateCollectionAsync(collectionName, ...)`) passes the collection name as a plain string argument to the Qdrant .NET client's typed gRPC API — not a textual query language. This is **not** susceptible to the same injection class; TMA's hedge ("likely extends there too") does not hold up under direct code inspection. Qdrant collection naming is out of scope for this finding.

**Remediation:**
- **Primary fix:** At `RegisterSchema` time, before any descriptor reaches a schema manager, validate `TypeName` and every property name against a strict allow-list (e.g. `^[A-Za-z][A-Za-z0-9]*$`), rejecting anything else with `InvalidArgument`. This closes both Finding #1's "redefine an existing type" vector's DDL-injection component and any brand-new-type registration attempt.
- **Architectural improvement:** Consider deriving table/column names from a hash or a server-generated identifier rather than directly from client input, decoupling the trust boundary entirely; at minimum, centralize identifier validation in one place (e.g., inside `SchemaBuilder.BuildDescriptor`) so both current and future schema managers inherit it automatically.
- **Defense-in-depth:** Run the application's Postgres/StarRocks roles with the minimum privilege needed for DDL (e.g., no `DROP`/superuser rights) as a blast-radius limiter independent of the input-validation fix.

## Finding #3: `Aggregate`/`GroupBy` expression fields are client-reachable despite source comments claiming otherwise, and their validation does not restrict non-identifier characters

**Vulnerability:** Injection (weakened input validation) — A03, plus a documentation/trust-assumption defect
**Severity:** Medium
**Confidence:** Confirmed (validation gap and reachability); High-not-Confirmed for full query-structure manipulation (see exploitability analysis)
**Attack Complexity:** Medium
**TMA cross-link:** Confirms and substantially refines TMA #F7

**Location:**
- File: `Iverson.Clients/Common/Proto/object_search.proto`, Lines: 176 (`AggregationSpec.expression`), 219 (`MetricSpec.expression`)
- File: `Iverson.StarRocks/StarRocksQueryBuilder.cs`, Lines: 182–189, 244–251 (`BuildAggregate`), 338, 345–350, 397–412 (`BuildGroupBy`/`BuildMetricExpr`)
- File: `Iverson.StarRocks/StarRocksPipelineBuilder.cs`, Lines: 24–32 (`TokenRx`/`DeriveWhitelist`), 320–334 (`DeriveColumn.Expr`'s stricter validation, for comparison)

**Risk & Exploit Path:**
`AggregationSpec.expression` and `MetricSpec.expression` are ordinary `string` fields on public protobuf messages (`AggregateRequest.aggregations[]`, `GroupByRequest.metrics[]`) — reachable by any caller with read access to the target entity type via the `Aggregate`/`GroupBy` RPCs. This directly contradicts the source code's own inline comments:
> `// spec.Expression, when set, is raw StarRocks SQL supplied by a trusted server-side caller (e.g. TPC-H DSL translation) and is spliced directly into the aggregate function — NOT user input, no escaping is performed.`

This assumption is empirically false relative to the actual proto contract — there is no server-side-only code path that constructs these fields; the client sets them directly. The validation that *does* exist (`TokenRx.Matches(expression)`, checking each identifier-shaped substring against `DeriveWhitelist` ∪ the entity's real column names) only inspects tokens matching `[A-Za-z_][A-Za-z0-9_]*`. Every other character — whitespace, parentheses, commas, arithmetic/comparison operators, string-literal quotes, and critically, comment sequences (`--`, `/* */`) — is never scanned or restricted, and the *original, unmodified string* (not a reconstructed safe version) is what gets spliced into the final SQL.

**Exploitability analysis (traced, not assumed):** I traced both call sites' full SQL templates:
```csharp
// StarRocksQueryBuilder.cs:247 (BuildAggregate) — Expression sits inside the SELECT list, BEFORE {from}{wc}{hc}
AggregationKind.Avg => $"SELECT AVG({spec.Expression ?? Quote(col)}) AS metric_val {from}{wc}{hc}",
```
```csharp
// StarRocksQueryBuilder.cs:338 (BuildGroupBy) — same property: selectCols (containing metricExprs) precede {from}{wc}
var sql = $"SELECT {string.Join(", ", selectCols)} {from}{wc} GROUP BY {...}{hc}{oc} LIMIT {limit}";
```
Because each query is built as a single-line interpolated string and the vulnerable `Expression` is always spliced into the SELECT list — which, by SQL syntax, must precede `FROM`/`WHERE` — an inline `--`/`/* */` comment injected via `Expression` comments out everything from that point to the end of the line, **including the `FROM` clause itself and the row-ownership `WHERE` predicate together**. There is no way to strip only the trailing ownership filter while preserving a working table reference using this specific gap; the attacker cannot selectively bypass row-ownership filtering while still returning real aggregated data through this vector alone. Separately, because `FROM`, `WHERE`, `SELECT`, table names, and non-whitelisted keywords are themselves rejected as unauthorized "fields" if referenced as identifier tokens, the attacker also cannot reconstruct an alternate query referencing other tables/columns by name.

**Net assessment:** the validation approach is genuinely unsound (regex-token allow-listing does not validate the full string), and the source comment's trust assumption is factually wrong given the real reachable surface — both are real defects worth fixing. However, direct verification of the two call sites shows the *worst-case impact* (bypassing row-ownership filtering to read other tenants' data) is not achievable through this specific gap today, because of the templates' clause ordering. This is a **latent** risk: a future refactor of either query builder (e.g., toward the CTE-based structure `StarRocksPipelineBuilder` already uses elsewhere, which happens to have the *stricter* blacklist) could silently reintroduce a fully exploitable path without anyone touching the `Expression`-handling code itself.

**Evidence / Trace:**
```csharp
// StarRocksPipelineBuilder.cs:24-32 — the weaker validation used by AggregationSpec/MetricSpec.Expression
internal static readonly Regex TokenRx = new("[A-Za-z_][A-Za-z0-9_]*", RegexOptions.Compiled);
internal static readonly HashSet<string> DeriveWhitelist = new(StringComparer.OrdinalIgnoreCase)
{ "SUM", "AVG", "MIN", "MAX", "COUNT", "OVER", "PARTITION", "BY", "ORDER",
  "ASC", "DESC", "COALESCE", "NULLIF", "ROUND", "ABS", "AND", "OR", "NOT", "NULL" };
```
```csharp
// StarRocksPipelineBuilder.cs:320-334 — the STRICTER validation used by DeriveColumn.Expr, for contrast
// (not shown here in full — additionally hard-blocks `; ' ` -- /* */` outright, per repo comments)
```

**Remediation:**
- **Primary fix:** Correct the misleading comments (they will mislead a future maintainer into further weakening validation, believing the field is trusted). Align `Expression`'s validation with `DeriveColumn.Expr`'s stricter denylist (block `; ' `` -- /* */` outright, not just identifier-token checking).
- **Architectural improvement:** Replace regex-token allow-listing with an actual small expression parser/AST for both paths, so "is this string safe" is answered structurally rather than by scanning for word-shaped substrings.
- **Defense-in-depth:** Given the latent risk tied to template ordering, add a regression test that asserts row-ownership filtering survives an adversarial `Expression`/`expression` value containing `--`/`/* */`, so a future query-builder refactor can't silently reopen this path unnoticed.

## Finding #4: Recurrence of a previously-fixed log-forging vulnerability class across gRPC entry points

**Vulnerability:** Log Injection / Improper Output Neutralization for Logs — Category 9 (Error Handling & Information Leakage), CWE-117
**Severity:** Low
**Confidence:** Confirmed
**Attack Complexity:** Low
**TMA cross-link:** Net-new (not in TMA scope — TMA does not do code-level vuln-hunting)

**Location:**
- File: `Iverson.Api/Grpc/ObjectMappingGrpcService.cs`, Lines: 49, 137, 178, 244, 308
- File: `Iverson.Api/Grpc/ObjectPersistenceGrpcService.cs`, Lines: 52, 124
- File: `Iverson.Api/Grpc/ObjectRetrievalGrpcService.cs`, Lines: 26, 62
- File: `Iverson.Api/Grpc/ObjectSearchGrpcService.cs`, Lines: 56, 160, 231, 289, 346, 405
- Related (the prior, narrower fix): `Iverson.Api/Reconciliation/ReconciliationService.cs:49` (commit `4b85354`, 2026-07-15)

**Risk & Exploit Path:**
On 2026-07-15, a CodeQL `cs/log-forging` alert was fixed at exactly one call site: `ReconciliationService.cs`, reached via the Operator-gated `POST /admin/reconcile/{typeName}` HTTP route. The fix (`typeName.ReplaceLineEndings("")`) was correct but scoped to that single flagged instance rather than the underlying pattern — client-supplied `TypeName` reaching `ILogger` templates unsanitized recurs at no fewer than 15 other call sites across all four gRPC services, none of which strip line endings before logging. `request.TypeName` is the exact same field implicated in Findings #1/#2 — the same value an attacker would already be crafting to exploit those findings can simultaneously carry embedded newlines/carriage returns to make one log entry render as several when viewed as plain text, complicating exactly the kind of incident-response log review that would otherwise catch an in-progress F1/F2 exploitation attempt.

**Evidence / Trace:**
```csharp
// ObjectMappingGrpcService.cs:49 — one of 15+ unfixed sites, same pattern as the one that was fixed
_logger.LogInformation("[RegisterSchema] root={Type} dependents={Deps}",
    request.RootType?.TypeName, request.Dependents.Count);
    // ← VULNERABLE: TypeName not passed through .ReplaceLineEndings("") the way
    //   ReconciliationService.cs's typeName now is
```
```csharp
// ReconciliationService.cs:49 — the fix that established the correct pattern, not generalized elsewhere
logger.LogInformation("[Reconcile] Re-projected {Count} {Type} records to Kafka",
    count, typeName.ReplaceLineEndings(""));
```

**Remediation:**
- **Primary fix:** Apply `.ReplaceLineEndings("")` (or a shared sanitizing helper) to every `TypeName`/user-supplied value reaching a log template across the four gRPC services — grep for `request.TypeName` in `LogInformation`/`LogWarning`/`LogDebug` calls and fix all matches in one pass.
- **Architectural improvement:** Wrap this in a small `LogSafe(string)` extension or a custom `ILogger` formatter applied consistently, so the fix doesn't require remembering to call `.ReplaceLineEndings()` at every future call site — this is exactly the kind of recurrence a single narrowly-scoped fix predictably produces.
- **Defense-in-depth:** Re-run CodeQL's advanced-setup C# build (already enabled per commit `4b85354`'s own description) and confirm it now flags these additional sites — if it doesn't, that's worth understanding, since it means the query's reachability analysis is narrower than manual review found here.

## Finding #5: Qdrant has no authentication in any deployment target (StarRocks corrected — see note)

**Vulnerability:** Security Misconfiguration / Broken Access Control — A05/A01
**Severity:** High (Qdrant, both targets) / Low (StarRocks, compose only)
**Confidence:** Confirmed
**Attack Complexity:** Low (for any actor already positioned on the relevant network segment)
**TMA cross-link:** Confirms TMA #F4 (as corrected — see note)

> **Correction to this finding's original scope:** the first draft of this finding (and TMA's underlying F4) incorrectly claimed StarRocks has no authentication in *either* deployment target. Direct code verification during remediation planning found this is wrong for kind/cloud: `deploy/helm/iverson/charts/starrocks/templates/job-create-user.yaml` provisions a dedicated `iverson_app` user with a Helm-generated password and scoped `GRANT`s (`SELECT/INSERT/UPDATE/DELETE`, `CREATE TABLE/DROP/ALTER` on the database, `OPERATE ON SYSTEM` for health checks, `CREATE DATABASE ON CATALOG` for `RegisterSchema`), and `Iverson.Api`'s deployment already wires that generated password into `ConnectionStrings__StarRocks` (`charts/api/templates/deployment.yaml:85-91`). StarRocks in kind/cloud is **not** a gap. StarRocks in **compose** genuinely has none (root, empty password) — consistent with that target's already-accepted dev-only posture, and now scored separately below at Low rather than folded into a High finding it doesn't belong in. Qdrant's "no authentication in either target" claim is unaffected by this correction — independently confirmed (no API-key config anywhere in `Iverson.Vector`, the Qdrant Helm chart, or docker-compose.yml).

**Location:**
- File: `Iverson.Server/deploy/helm/iverson/charts/qdrant/**` (no API-key config anywhere)
- File: `Iverson.Server/Iverson.Vector/ServiceCollectionExtensions.cs:13` (`new QdrantClient(host, port)` — no apiKey argument)
- File: `Iverson.Server/deploy/helm/iverson/templates/networkpolicies.yaml` (sole control in kind/cloud, for Qdrant)
- File: `Iverson.Server/docker-compose.yml` (flat network, sole control in compose, for both Qdrant and StarRocks)
- Contrast (StarRocks kind/cloud, confirmed NOT vulnerable): `charts/starrocks/templates/job-create-user.yaml`, `charts/api/templates/deployment.yaml:85-91`

**Risk & Exploit Path:**
Qdrant is configured with zero authentication in every deployment target this repo defines. In kind/cloud, reachability is gated exclusively by Kubernetes `NetworkPolicy`; in docker-compose, by nothing (all containers share one flat Docker network with published ports). Any workload capable of reaching Qdrant's ClusterIP — a compromised sidecar or adjacent pod, a `NetworkPolicy` misconfiguration during a future Helm change, or literally any container in the compose topology — obtains full, unauthenticated read/write access, completely bypassing every row/field authorization control described in Findings #1–#3 and TMA §5(B). Postgres and (in kind/cloud) StarRocks both have real credential-based authentication as an independent second layer behind `NetworkPolicy`; Qdrant has none in either target, making it the actual "weakest link" for data-store compromise — not the StarRocks-and-Qdrant pairing originally claimed.

**Evidence / Trace:** Confirmed via direct inspection of the Qdrant Helm chart (no Secret template, no API-key env var anywhere) and `Iverson.Vector/ServiceCollectionExtensions.cs:13`'s `QdrantClient(host, port)` constructor call, which takes no credential argument — this is an absence-of-control finding, not a single vulnerable line.

**Remediation:**
- **Primary fix:** Enable Qdrant's API-key authentication in the kind/cloud Helm chart (a Secret + `QDRANT__SERVICE__API_KEY` env var on the StatefulSet), and pass the resulting key into `AddQdrant`/`QdrantClient`'s constructor (`Qdrant.Client` 1.18.1 supports an `apiKey` parameter).
- **Architectural improvement:** Treat this as a required constraint before any production-shaped deployment, not an optional hardening step — `NetworkPolicy` is Qdrant's only control today, with no backstop.
- **Defense-in-depth:** For compose (both Qdrant and StarRocks, explicitly dev-only), at minimum document that it must never be exposed beyond localhost.

## Finding #6: Compose-only OAuth2 client secrets and user passwords are committed to source control

**Vulnerability:** Cryptographic Failures & Data Exposure — A02 (Hardcoded Secrets)
**Severity:** Medium
**Confidence:** Confirmed
**Attack Complexity:** Trivial (secrets are in the git history)
**TMA cross-link:** Confirms TMA #F5

**Location:**
- File: `Iverson.Server/deploy/helm/iverson/charts/authentik/blueprints/compose-only/service-clients.yaml`

**Risk & Exploit Path:**
Static, human-readable OAuth2 client secrets and default user passwords for the docker-compose Authentik target are committed to git, clearly labeled `dev-only-not-for-production...`. Blast radius is bounded to whatever consumes this compose file — the kind/cloud path generates real per-release secrets via Helm's `randAlphaNum`, persisted with `helm.sh/resource-policy: keep`, and does not share this exposure. This finding technically satisfies the SQS hard-gate criterion ("hardcoded secrets... in source code") regardless of severity scoring — see the SQS section below.

**Remediation:**
- **Primary fix:** No code change is strictly required if compose is confirmed to be exclusively local/developer-machine-only in every real usage of this repo. Confirm and document that assumption explicitly (e.g., in the compose file's header comment or a CONTRIBUTING note).
- **Architectural improvement:** If compose is ever used in a shared or CI environment, move these to generated-at-deploy-time secrets following the kind/cloud pattern already established in this same repo.
- **Defense-in-depth:** Rotate these specific values if there's any chance the compose target has ever been reachable outside a fully isolated local machine.

## Finding #7: Kafka-projected stores trust event payload content (including ownership values) without re-authorization

**Vulnerability:** Broken Access Control / Implicit Trust — A01
**Severity:** Medium
**Confidence:** Confirmed
**Attack Complexity:** N/A standalone (amplifier — see Pass 3)
**TMA cross-link:** Confirms TMA #F3

**Location:**
- File: `Iverson.Api/Consumers/IntelligenceStoreConsumer.cs`, Lines: 138–140
- Related: `Iverson.Events/EntityEvent.cs:20-28` (no identity/owner claim in the event schema itself)

**Risk & Exploit Path:**
`EntityEvent`s are plain JSON with no signing or integrity check. `IntelligenceStoreConsumer` reads the event payload's own field values — including whatever will become the row's ownership value for Qdrant's read-time filtering — verbatim, with no re-derivation from or comparison against the authoritative Postgres row. This is not independently reachable externally (Kafka has no ingress in either target), but it means the two projected stores' ownership filtering is only as trustworthy as whatever produced the original event — directly amplifying Findings #1/#2: an attacker who achieves an elevated write identity via those findings can stamp arbitrary ownership data that propagates into StarRocks/Qdrant unchecked.

**Remediation:**
- **Primary fix:** Re-derive or re-validate the owner value at projection time against the authoritative Postgres row rather than trusting the event payload verbatim, closing this as an independent control rather than relying solely on write-time enforcement.
- **Architectural improvement:** Consider adding a minimal integrity mechanism (e.g., HMAC over the event payload using a key only the API process holds) if cross-service trust boundaries are expected to widen in the future (e.g., additional consumers written by other teams).

## Finding #8: All 5 Dockerfiles run as root; mitigated in kind/cloud, not in compose

**Vulnerability:** IaC/Container Misconfiguration — Category 10
**Severity:** Low
**Confidence:** Confirmed
**Attack Complexity:** N/A (defense-in-depth gap, not independently exploitable)

**Location:**
- Files: `Iverson.Api/Dockerfile`, `Iverson.Sql/Dockerfile`, `Iverson.StarRocks`... (`Iverson.Vector/Dockerfile`, `Iverson.Events/Dockerfile`, `Iverson.Launcher/Dockerfile`) — none declare a `USER` directive (5 similar instances)
- Related (the mitigating control): `deploy/helm/iverson/charts/api/templates/deployment.yaml:54-56` (`securityContext: runAsNonRoot: true, runAsUser: 1000`)

**Risk & Exploit Path:**
None of the 5 first-party Dockerfiles set a non-root `USER`, so the built images default to root. **This is verified to be mitigated in the kind/cloud deployment target**: the Helm chart's pod `securityContext` enforces `runAsNonRoot: true` / `runAsUser: 1000` at the Kubernetes level, overriding the image's default at runtime — confirmed directly in the deployment template, not assumed from a CI comment referencing it. The residual gap is narrower than a naive "containers run as root" finding would suggest: it applies to (a) docker-compose, which sets no equivalent `user:` override for any first-party service, and (b) any ad-hoc `docker run` of these images outside the Helm chart's control.

**Remediation:**
- **Primary fix:** Add a non-root `USER` directive to each Dockerfile directly, so the image is safe by default regardless of the runtime environment's own security context — removes the current sole dependency on the Helm chart remembering to set it.
- **Defense-in-depth:** Add an equivalent `user:` override to docker-compose.yml's first-party services for parity with the kind/cloud posture.

## Finding #9: Several docker-compose image references use unpinned `:latest` tags

**Vulnerability:** Vulnerable Components & Supply Chain — A06
**Severity:** Low
**Confidence:** Confirmed
**Attack Complexity:** N/A

**Location:**
- File: `Iverson.Server/docker-compose.yml` — `starrocks/allin1-ubuntu:latest` (line 25), `ollama/ollama:latest` (line 56), `curlimages/curl:latest` (line 70), `jaegertracing/all-in-one:latest` (line 118) (4 similar instances)

**Risk & Exploit Path:**
These four third-party image references float on `:latest` rather than a pinned version, meaning behavior can silently change on any redeploy and there is no protection against a compromised or altered upstream `latest` tag. ## 1. Executive Summary

Iverson.Server's **data-plane authorization** — the row/field-level `RowFieldAuthorizationEvaluator` enforced across every CRUD/retrieval/search RPC — is genuinely well-built: it fails closed by default, correctly re-checks per-row ownership even in batch endpoints (`GetMany`), independently validates a second JWT scheme for acting-user identity rather than trusting a claim inside the primary token, and consistently masks/rejects disallowed fields on read and write. This is not a codebase with a sloppy or absent authorization model.

The problem is that this well-built model has an enormous, unguarded side door: **`RegisterSchema` — the operation that *defines* those authorization rules in the first place — has no authorization check of its own** (Finding #1), and the identifiers it accepts flow unescaped into live DDL against two of the three data stores (Finding #2). Together these mean any caller holding a JWT for *any* of the system's four configured audiences — including ordinary users of the human-facing web application, not just service credentials — can grant themselves full bypass access to any entity type, or execute arbitrary DDL, with a single unauthenticated-by-role gRPC call. This is the dominant risk in the system and should be treated as a release blocker.

A second tier of findings (Qdrant having no authentication in any environment — StarRocks turned out to already have real authentication in kind/cloud, corrected below in Finding #5 — Kafka-projected stores trusting event payloads without re-checking authorization, and a genuinely unsound — though currently structurally bounded — validation gap in the Search/Aggregate expression DSL) reflects a consistent pattern: authorization decisions made once, correctly, at one layer, without an equivalent check at every layer data subsequently flows through. None of these is independently as severe as Findings #1/#2, but they compound with them (Pass 3, below) and are worth fixing as part of the same effort rather than treated as separate, lower-priority work.

The remaining findings (log-forging recurrence, container/IaC hygiene, unpinned dev-compose image tags) are real but low-severity, and mostly reflect a fix that was applied narrowly rather than generalized (Finding #4 is the clearest example — a real vulnerability class was correctly identified and fixed at one site in this repo's very recent history, and recurs at fifteen more).

**Production readiness: not ready**, specifically because of Findings #1 and #2. Nothing else in this review would independently block a release; those two would.

## 4. Positive Security Observations

- **Row/field authorization fails closed correctly, in two independent ways.** `RowFieldAuthorizationEvaluator.Evaluate` (`Authorization/RowFieldAuthorizationEvaluator.cs:11-15`) denies both when a schema has no `Authorization` rules configured *and* when the acting-user identity is absent — verified directly in code, not assumed. Omitting the acting-user header is a hard deny for any schema that requires it, not a silent bypass.
- **Batch retrieval re-checks ownership per row, not just once for the whole batch.** `ObjectRetrievalGrpcService.GetMany` (`Grpc/ObjectRetrievalGrpcService.cs:75-104`) correctly evaluates the authorization decision once (appropriate, since it doesn't depend on row content) but then independently checks each individual row's owner value against it — this is not the IDOR pattern its shape initially suggested; it was directly traced and confirmed sound.
- **Acting-user identity is a genuinely independent second credential, not a spoofable claim.** The `ActingUser` JWT scheme (`Program.cs:107-132`) is separately signature-validated against its own Authority/audience configuration; a caller cannot forge an acting-user identity by manipulating only the primary service credential.
- **Search/Aggregate/GroupBy's column and property references are properly allow-listed.** `StarRocksQueryBuilder.ResolveColumn` rejects unknown or unauthorized field references outright across `WHERE`, `GROUP BY`, `ORDER BY`, and `HAVING` clause construction — the only weaker spot found was the narrower `Expression` fields (Finding #3), not the general query-building path.
- **`MapOpenApi()` (Swagger) is correctly gated behind `IsDevelopment()`, and gRPC's `EnableDetailedErrors` is left at its safe (disabled) default** — no stack-trace/internal-exception leakage to clients was found.
- **JWT validation uses Authority-driven JWKS resolution with no static `IssuerSigningKey` override** — the classic RS256/HS256 algorithm-confusion attack class does not apply here, since there is no code path that would accept a symmetric key for a token whose `alg` header claims asymmetric signing.

## 5. Prioritized Remediation Roadmap

1. **Finding #1 + Finding #2 together — `RegisterSchema` authorization gate + identifier allow-listing.**
   Why first: this is the dominant, chained risk in the system — a single authenticated call from any of 4 broad audiences currently yields full authorization bypass and/or DDL execution. Both fixes touch the same method and are naturally done together.
   Effort: **Moderate** (an authorization policy check plus a regex validation are individually small, but need a decision on the right policy shape and a pass to confirm no legitimate current caller breaks).
   Owner: Backend / Security Team (joint sign-off recommended given the severity).

2. **Finding #5 — Qdrant authentication.**
   Why: second-highest severity, and independently sufficient to bypass everything Finding #1/#2's fix would otherwise close, for any actor with network access to Qdrant. (StarRocks needed no fix in kind/cloud once its existing `iverson_app` credential was found; only its compose posture remains an accepted dev-only gap.)
   Effort: **Moderate** (Helm chart change to generate + mount an API key on the Qdrant StatefulSet, plus a small `Iverson.Vector` client change to pass it — narrower than initially estimated once StarRocks was found to already be covered).
   Owner: DevOps + Backend (shared).

3. **Finding #6 — compose secrets in git.**
   Why: independently hard-gates the SQS regardless of numeric severity; cheap to resolve or explicitly accept-and-document.
   Effort: **Quick Win** (confirm compose's local-only usage and document it, or rotate + move to generated secrets if not).
   Owner: DevOps.

4. **Finding #3 — expression validation + comment correction.**
   Why: real defect, currently structurally bounded, but latent and cheap to fix now versus discovering it's no longer bounded after a future query-builder refactor.
   Effort: **Quick Win to Moderate** (denylist alignment is quick; an AST-based replacement is moderate).
   Owner: Backend.

5. **Finding #4 — log-forging recurrence.**
   Why: lowest severity here, but cheapest fix-to-impact ratio, and directly relevant to detecting exploitation of #1/#2 should it occur before they're fixed.
   Effort: **Quick Win** (mechanical fix across ~15 call sites, ideally centralized into a shared helper to prevent the next recurrence).
   Owner: Backend.

Findings #7, #8, #9 are worth fixing as part of the same remediation pass (all Quick Win-to-Moderate) but don't independently change the release-readiness verdict the way #1/#2/#5/#6 do.
